# 首页瓶子弹窗收件箱设计

## 用户体验

首页右上角保留轻量按钮，但语义改为用户瓶子数量。点击后打开覆盖在海面上的单个弹窗，桌面端采用左右布局：左侧为瓶子列表，右侧为内容或聊天。移动 H5 使用同一个弹窗容器，窄屏时列表和内容上下排列。

捡瓶成功后不再跳转瓶子详情页，而是在动画结束后打开弹窗并选中刚捡到的瓶子。无论瓶子来源是用户还是 AI，内容都必须显示在右侧区域。

## 数据来源

新增 `GET /v1/bottles/mine`，返回当前用户可见的瓶子摘要：

- 用户捡到且未删除的瓶子。
- 用户扔出且未被删除的瓶子。
- 如果瓶子已经开启会话，返回对应 `conversationId`。

列表只提供摘要和会话引用，正文仍通过 `GET /v1/bottles/:id` 加载，聊天仍通过 `GET /v1/chats/:id/messages` 加载，避免复制详情与聊天逻辑。

## 状态处理

弹窗选中列表项时：

- 有 `conversationId` 时加载并显示聊天窗口。
- 没有 `conversationId` 且为捡到的瓶子时加载瓶子详情并显示回复框。
- 没有 `conversationId` 且为扔出的瓶子时加载瓶子详情并显示瓶子内容。

删除捡到的瓶子或删除聊天后刷新列表和右上角数量。错误通过现有 toast 明确提示，不做静默兜底。
