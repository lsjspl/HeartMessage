# 瓶子与聊天闭环设计

## 数据模型

当前已有 `daily_quotas`、`bottles`、`bottle_pickups`、`conversations` 和 `messages` 表。实现阶段优先复用这些表，新增字段只用于真实业务缺口：

- `conversations.deleted_by_participant_a_at`：A 方删除会话时间。
- `conversations.deleted_by_participant_b_at`：B 方删除会话时间。
- `messages.read_at`：接收方读取时间，第一阶段可为空。

删除关系不物理删除消息，避免影响另一方历史记录。用户删除会话后，自己不再看到该会话，也不能继续收发该会话消息；另一方后续发送消息时 API 必须显式拒绝，不做静默投递。

## 每日额度

额度按服务器本地日期字符串 `YYYY-MM-DD` 统计。第一阶段使用 Worker 运行时的当前时间计算日期；后续如果需要严格按用户时区，可在用户资料中增加时区字段。

- `GET /v1/bottles/quota` 返回当天已用和剩余额度。
- `POST /v1/bottles/throw` 在同一个事务中检查并递增扔瓶子次数。
- `POST /v1/bottles/pick` 在同一个事务中检查并递增捡瓶子次数。

超过额度时返回结构化错误，不返回假成功。

## 扔瓶子

瓶子内容由共享 Zod schema 校验。创建后状态直接为 `floating`，`expiresAt` 设置为当天 23:59:59.999。内容审核能力后续单独接入；当前阶段不保留 `reviewing` 旧兜底。

## 捡瓶子

捡瓶子只从 `floating` 且 `expiresAt > now` 的瓶子中选择，排除当前用户自己扔出的瓶子。为了 D1 可实现和结果可控，第一阶段按 `createdAt ASC` 获取最早可捞瓶子；后续可以增加随机权重。

成功捡到后：

1. 创建 `bottle_pickups`。
2. 将瓶子状态改为 `picked` 并写入 `pickedAt`。
3. 递增当天捡瓶子次数。

如果没有瓶子，返回 `NO_BOTTLE_AVAILABLE`。AI 自动补瓶后续由 AI 变更接入，当前阶段不静默生成假瓶子。

## 回复与聊天

用户只能回复自己捡到且未删除的瓶子。回复成功后创建会话，参与者 A 为瓶子作者，参与者 B 为捡瓶用户，首条消息由回复者发送。若会话已存在，则返回已有会话并追加消息时必须遵循同一会话状态。

聊天消息只能由会话参与者发送。会话处于删除或封禁状态时拒绝发送。

## 前端接入

用户端移除固定 demo 跳转：

- 首页加载真实额度，点击捡瓶子调用接口。
- 扔瓶子页提交真实内容并处理额度错误。
- 瓶子详情页展示接口返回的瓶子信息，支持删除和回复。
- 聊天列表和详情页从 API 获取数据，发送消息后刷新。

## 后台

后台瓶子列表第一阶段展示最近 100 条瓶子，包含作者、内容摘要、来源、状态、创建时间和过期时间。后台鉴权仍沿用当前骨架约束，正式管理员登录后续单独变更补齐。
