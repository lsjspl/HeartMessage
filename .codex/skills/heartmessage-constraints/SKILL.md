# HeartMessage 项目硬约束

## 核心原则

在 HeartMessage 项目中执行任何代码、文档、配置、OpenSpec、数据库、Cloudflare 或部署相关任务时，先应用本 skill。以下规则是硬约束；除非用户明确、具体地要求临时例外，否则不得绕过。

## 必守规则

1. 单个文件严禁超过 1500 行。
   - 新建文件必须控制在 1500 行以内。
   - 修改文件前先评估目标文件行数和改动后规模。
   - 如果文件已经接近或超过 1500 行，不得继续往里堆功能；必须拆模块、拆组件、拆 service、拆 spec 或拆文档。
   - 如果工具生成的文件可能超过 1500 行，必须明确说明并暂停询问，不能静默接受。

2. 所有功能必须模块化，尽量减少耦合。
   - 按领域拆分：认证、用户、瓶子、聊天、AI、后台、数据库、上传、日志等边界要清楚。
   - 路由只做参数校验和调用服务；业务逻辑放 service；共享类型放 shared；数据库结构放 db。
   - 不要把多个无关功能混进一个文件。
   - 不要为了省事在页面、路由或工具文件里塞跨领域逻辑。

3. 所有文档必须使用中文。
   - README、OpenSpec、设计文档、任务清单、注释型说明、项目规范都必须中文。
   - OpenSpec 必须保留校验器需要的固定关键字，例如 `Purpose`、`Requirements`、`Requirement`、`Scenario`、`MUST`、`WHEN`、`THEN`、`AND`；其余正文必须中文。
   - 如果引入第三方英文原文，只能作为引用或链接，不得把项目文档主体写成英文。

4. 严禁私自操作生产环境。
   - 不得擅自执行生产部署、生产数据库迁移、生产数据删除、生产 KV/R2/Queue 操作、生产密钥修改。
   - 禁止在未明确确认的情况下运行 `wrangler deploy`、`wrangler d1 ... --remote`、生产环境脚本或指向生产资源的命令。
   - 如确实需要生产操作，必须先列出目标环境、资源、命令、影响范围和回滚方式，并获得用户明确同意。

5. 严禁静默兜底、旧代码兜底、旧逻辑不删除。
   - 不允许“新逻辑失败后悄悄走旧逻辑”。
   - 不允许保留无人使用的旧分支、旧函数、旧配置、旧页面，只为了心理安全。
   - 替换实现时必须删除或停用旧逻辑；如果确实需要兼容期，必须写入 OpenSpec 或任务说明，并明确期限、开关和删除计划。
   - 错误必须显式暴露，不能吞异常、不能假成功、不能返回看似正常但实际降级的数据。

6. 新功能必须 OpenSpec 先行。
   - 没有 proposal、design、tasks 和 spec delta，不允许直接实现业务功能。
   - 一个 OpenSpec change 只做一个明确功能；如果范围变大，必须拆成多个 change。
   - 已完成的实现必须同步更新 tasks 状态；不能让文档和代码长期不一致。

7. 不允许跳过验证。
   - 改代码后必须运行相关校验，例如 `pnpm spec:validate`、`pnpm typecheck`、对应应用 build 或测试。
   - 如果校验无法运行，必须在最终回复中明确说明原因和风险。
   - 不能声称“应该没问题”来代替验证结果。

8. 不允许私自引入新依赖。
   - 新增 npm 包、SDK、UI 库、状态库、工具库前，必须说明为什么现有技术栈不够用。
   - 能用现有依赖、标准库或项目内工具解决时，不得为了方便加新依赖。
   - 引入依赖后必须更新对应 package，并运行安装和校验。

9. 不允许做无关重构。
   - 只修改当前 OpenSpec 或用户请求覆盖的范围。
   - 看到旧代码不顺眼不能顺手大改。
   - 如发现必须先重构才能安全实现，必须说明原因并把重构纳入任务范围。

10. 不允许私自执行 git 发布链路。
    - 未经用户明确要求，不得执行 `git add`、`git commit`、`git push`、创建分支、创建 PR 或发布版本。
    - 可以读取 git 状态和 diff 来确认影响范围，但不能擅自改变仓库历史或远端状态。

11. 严禁提交密钥和敏感配置。
    - API Key、微信 secret、Cloudflare token、数据库 ID、生产环境变量不得写入代码、文档或示例中的真实值。
    - 示例值必须使用 `replace-with-*`、`local-*` 或明显的占位符。
    - `.env`、本地日志、构建产物、临时文件不得提交。

12. 数据库变更必须有 migration。
    - 改 Drizzle schema 时，必须同步提供 D1 migration。
    - 涉及数据删除、字段迁移、兼容旧数据时，必须写清影响范围和回滚思路。
    - 未经用户明确确认，不得对远程 D1 执行 migration。

13. API 契约必须放在共享包。
    - 前后端共用的 DTO、枚举、常量、Zod schema 必须放在 `packages/shared`。
    - 不允许前端和后端各写一份接口类型。
    - API 错误必须保持结构化：`{ ok: false, error: { code, message } }`。

14. AI 供应商不得写死在业务里。
    - 瓶子、聊天、后台配置等业务代码不得硬编码 OpenAI、DeepSeek 或其他具体模型供应商。
    - AI 调用必须通过 `packages/ai` 和后台模型配置选择用途模型。
    - AI 失败必须显式报错或进入可观测任务状态，不能静默换模型或假装成功。

15. 生产安全必须显式处理。
    - 生产 CORS 不能使用 `*`，必须使用白名单。
    - 日志不能记录 token、API Key、完整 openid、生产密钥、用户隐私内容或完整聊天正文。
    - `/admin/*` 正式功能必须鉴权；骨架阶段留下未鉴权接口时，必须在 OpenSpec 或任务中标记为待补。

16. 生成文件不进仓库。
    - `dist`、`.wrangler`、`unpackage`、日志、临时文件、截图中间产物等不得进入 git。
    - 如果确实需要提交生成物，必须由用户明确要求，并说明原因。

17. 前端平台优先级必须按当前产品阶段执行。
   - 第一优先级是 Web。
   - 第二优先级是 H5。
   - 不得擅自把移动端优先作为默认策略。

18. 删除旧逻辑必须同步清理引用。
    - 删除旧实现时，必须同步清理路由、导出、类型、文档、测试和配置引用。
    - 不允许留下“看起来没人用但还在导出”的死代码。
    - 如果无法立即删除引用，必须写明阻塞原因和后续删除任务。

## 执行检查

修改前：

- 确认本次任务影响哪些模块，先选择最小边界。
- 检查将要修改或新增的文件是否会超过 1500 行。
- 确认是否涉及生产环境；如果涉及，先停止并请求明确确认。
- 确认是否会引入 fallback、兼容旧逻辑或保留旧实现；如果会，必须重新设计或写明删除计划。
- 确认是否需要先写或更新 OpenSpec；新功能必须先写规格。
- 确认是否需要新增依赖；如需要，先说明理由。

修改中：

- 优先新增小模块、小 service、小 schema，而不是扩大已有大文件。
- 删除被替代的旧逻辑。
- 文档、OpenSpec 和说明全部中文。
- 对错误使用明确失败路径，不做静默吞错。
- 共享 API 契约放 `packages/shared`。
- 数据库 schema 和 migration 同步维护。
- AI 供应商选择走配置和 `packages/ai`，不得写死在业务里。

完成前：

- 运行必要校验，例如 `pnpm spec:validate`、`pnpm typecheck`、对应应用构建或测试。
- 检查新增/修改文件没有超过 1500 行。
- 在最终回复中说明是否存在例外、未验证项或需要用户确认的生产操作。
- 检查没有误提交密钥、生成物、日志或临时文件。
- 检查没有私自执行 git 提交、推送、PR 或生产操作。
